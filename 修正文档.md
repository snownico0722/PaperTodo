## 第一项：数据保存、图片资产与损坏恢复

### 中高风险：两个数据文件的备份代际不一致

`data.json` 和 `note-assets.json` 分别保存、分别轮换备份，没有共同版本号。启动时一旦 `data.json` 从较旧备份恢复，程序会立即按照旧状态裁剪较新的图片资产。启动清理 图片裁剪

建议：

- 两个文件保存共同的 revision。
- 从备份恢复时不要立即物理删除图片。
- 利用现有但尚未实际使用的 `OrphanedAt`，先标记孤儿，经过一次成功保存或下一次正常启动后再删除。
- 恢复时单独保留一份清理前的图片资产快照。

### 中风险：多图片导入不是原子操作，而且写盘量呈平方增长

`ImportImageFiles` 逐张调用导入，每张都会重新序列化并写入完整 `note-assets.json`；只有全部成功返回后，编辑器才把引用插入正文。批量导入 逐次保存

如果第二张图片损坏：

- 第一张已经写入图片库。
- 整批调用抛错。
- 第一张的 Markdown 引用不会插入。
- 产生持久化孤儿，并重复进行整库写盘。

建议改成：全部预处理和校验 → 一次容量检查 → 一次加入内存 → 一次写盘；任何一步失败则整批回滚。

## 第二项：启动入口、快捷键与退出事务

### 高风险：退出保存发生在编辑内容提交之前

退出首先保存状态，之后才关闭设置窗口和纸片窗口。但标题编辑框、设置里的外部 Markdown 扩展名等内容是在失焦或关闭时才提交到模型。退出顺序 标题提交

因此通过以下入口退出时，最后输入可能丢失：

- 全局“退出”快捷键。
- `--exit` 命令。
- 系统结束会话。
- 标题编辑框仍保持焦点时直接退出。

设置文本框关闭后甚至会再触发一次异步保存，但程序紧接着 `Environment.Exit`，无法保证任务完成。设置关闭逻辑

建议建立明确顺序：

1. 设置 `_isExiting` 并停止计时器。
2. `FlushPendingEditorsForExit()`：提交标题、设置文本框和其他延迟编辑。
3. 同步保存并取得成功结果。
4. 保存成功后才清理图片资产。
5. 再注销快捷键、关闭窗口和回收进程。

### 中风险：启动期间的单实例命令可能静默丢失

主实例完成控制器加载、纸片恢复和首次命令执行后，才启动命名管道监听；次实例只重试约 1.4 秒，失败后仍静默退出。次实例重试 监听启动位置

图片库较大、恢复纸片较多时，启动可能超过这个时间。此时：

- `--new-note`
- `--new-todo`
- `--exit`

都可能没有执行，但次实例仍以成功方式退出。

建议主实例取得 Mutex 后立即启动管道监听，先把命令放入队列，控制器就绪后再依次执行。

### 中风险：快捷键注册成功与配置落盘不是一个事务

保存快捷键时先修改系统注册和内存状态，再异步保存 `data.json`。应用快捷键

如果磁盘保存失败：

- 新快捷键当前会话已经生效。
- UI 显示已注册。
- 磁盘仍是旧配置。
- 重启后快捷键突然恢复旧值。

建议保留旧绑定快照；系统注册成功后同步持久化，持久化失败则恢复旧系统绑定和旧 `State.GlobalHotkeys`。

### 中低风险：注销失败会留下“幽灵快捷键”

`TryUnregisterGesture` 失败时会保留 Win32 注册，但上层忽略返回值并继续报告成功；该按键可能仍被系统拦截，却不再执行命令。注销路径

建议注销失败时让整批应用失败，或者直接销毁并重建快捷键专用 `HwndSource`，确保系统注册全部释放。





### 第三项：胶囊状态机、焦点与菜单

1. 中风险：快速反向折叠/展开仍会跳变

中断动画时虽然读取了当前 Width、Height 和透明度，但随后新动画仍从固定的胶囊/纸片端点和固定透明度开始：

- 中断处理
- 重新使用固定起点

因此快速点击展开、Escape 收回、再展开时，仍可能瞬间跳到完整端点再继续动画。

一次 Escape 提交并收回、图片菜单关闭后恢复编辑焦点，以及拖动预览不直接修改持久顺序，这几条主路径没有发现同等级问题。

### 第六项：Markdown、图片与外部编辑

1. 高风险：图片限制没有真正约束解码内存

目前图片会先 `ReadAllBytes`，再通过 `BitmapCacheOption.OnLoad` 完整解码，之后才检查尺寸。

另外位图缓存只限制为 96 个条目，不限制解码后的总字节数。连续缩放可为同一图片产生多个宽度缓存；4096×4096 位图单张就可能占约 64MB。

建议：

- 读取前增加输入文件大小硬上限；
- 先读取元数据和像素数，再按目标尺寸解码；
- 缓存改成按解码后字节数控制的 LRU，并对宽度分桶。

1. 中风险：外部图片导出会静默降级，临时文件也不会清理

`ConvertMarkdownForExternalEditor` 在目录创建或单张图片写入失败时，会保留内部 `i:` 地址；随后 `WriteExternalMarkdownFile` 仍照常打开文件，普通 Markdown 编辑器只会看到损坏图片。

同时 `%TEMP%\PaperTodo` 下的 Markdown 和图片目录没有过期清理，删除笔记后也会长期残留。

建议让导出返回结构化成功/失败结果，使用临时目录完整写完后再替换，并增加按时间清理旧导出的启动任务。

纯图片 Ctrl+V、图片右键焦点恢复、删除后撤销、跨笔记粘贴时克隆图片资产，这些主要路径本身实现合理。

本次是静态代码审查。发布前仍建议在 Windows 上重点手测“大图连续缩放/复制”和“外部图片导出”。
