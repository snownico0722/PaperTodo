## 第一项：数据保存、图片资产与损坏恢复

### 高风险：状态保存失败后仍会删除图片

退出流程先调用 `SaveNow(sync: true)`，但 `SaveNow` 会在内部吞掉异常、不给调用者失败结果；随后退出流程无条件执行图片清理。保存实现 退出顺序

例如：

1. 用户删除一张图片或整篇笔记。
2. `data.json` 因占用、权限或磁盘问题保存失败。
3. 图片清理仍按当前内存状态删除资产。
4. 下次启动读取旧 `data.json`，Markdown 引用仍在，但图片已经没了。

这是目前第一项最严重的问题。

建议让同步保存返回成功状态；只有确认 `data.json` 已落盘，才允许执行破坏性的 `TrackReferences`。保存失败时宁可留下孤儿图片，也不要删图。

### 中高风险：两个数据文件的备份代际不一致

`data.json` 和 `note-assets.json` 分别保存、分别轮换备份，没有共同版本号。启动时一旦 `data.json` 从较旧备份恢复，程序会立即按照旧状态裁剪较新的图片资产。启动清理 图片裁剪

建议：

- 两个文件保存共同的 revision。
- 从备份恢复时不要立即物理删除图片。
- 利用现有但尚未实际使用的 `OrphanedAt`，先标记孤儿，经过一次成功保存或下一次正常启动后再删除。
- 恢复时单独保留一份清理前的图片资产快照。

### 中风险：多图片导入不是原子操作，而且写盘量呈平方增长

`ImportImageFiles` 逐张调用导入，每张都会重新序列化并写入完整 `note-assets.json`；只有全部成功返回后，编辑器才把引用插入正文。批量导入 逐次保存

如果第二张图片损坏：

- 第一张已经写入图片库。
- 整批调用抛错。
- 第一张的 Markdown 引用不会插入。
- 产生持久化孤儿，并重复进行整库写盘。

建议改成：全部预处理和校验 → 一次容量检查 → 一次加入内存 → 一次写盘；任何一步失败则整批回滚。

### 中低风险：图片引用清理没有核对所属笔记

当前只统计全局有哪些 `imageId` 被引用，并未检查引用它的笔记是否等于资产的 `NoteId`。恢复异常或手工修改数据后，可能保留一个实际上无法在当前笔记渲染的错误资产。

建议按 `(NoteId, ImageId)` 组合追踪，而不是只用全局 `ImageId`。

### 第一项中确认正确的部分

- `_saveVersion + SemaphoreSlim` 能阻止旧异步快照覆盖新状态。
- `data.json` 从备份恢复后会保留失败主文件和所用备份，处理较稳健。
- 图片先同步写入资产库、再插入 Markdown，崩溃时通常只会留下孤儿，不会生成已保存但无资产的引用。
- 删除引用后保留资产到退出，确实支持本次会话内撤销

## 第二项：启动入口、快捷键与退出事务

### 高风险：退出保存发生在编辑内容提交之前

退出首先保存状态，之后才关闭设置窗口和纸片窗口。但标题编辑框、设置里的外部 Markdown 扩展名等内容是在失焦或关闭时才提交到模型。退出顺序 标题提交

因此通过以下入口退出时，最后输入可能丢失：

- 全局“退出”快捷键。
- `--exit` 命令。
- 系统结束会话。
- 标题编辑框仍保持焦点时直接退出。

设置文本框关闭后甚至会再触发一次异步保存，但程序紧接着 `Environment.Exit`，无法保证任务完成。设置关闭逻辑

建议建立明确顺序：

1. 设置 `_isExiting` 并停止计时器。
2. `FlushPendingEditorsForExit()`：提交标题、设置文本框和其他延迟编辑。
3. 同步保存并取得成功结果。
4. 保存成功后才清理图片资产。
5. 再注销快捷键、关闭窗口和回收进程。

### 中风险：启动期间的单实例命令可能静默丢失

主实例完成控制器加载、纸片恢复和首次命令执行后，才启动命名管道监听；次实例只重试约 1.4 秒，失败后仍静默退出。次实例重试 监听启动位置

图片库较大、恢复纸片较多时，启动可能超过这个时间。此时：

- `--new-note`
- `--new-todo`
- `--exit`

都可能没有执行，但次实例仍以成功方式退出。

建议主实例取得 Mutex 后立即启动管道监听，先把命令放入队列，控制器就绪后再依次执行。

### 中风险：快捷键注册成功与配置落盘不是一个事务

保存快捷键时先修改系统注册和内存状态，再异步保存 `data.json`。应用快捷键

如果磁盘保存失败：

- 新快捷键当前会话已经生效。
- UI 显示已注册。
- 磁盘仍是旧配置。
- 重启后快捷键突然恢复旧值。

建议保留旧绑定快照；系统注册成功后同步持久化，持久化失败则恢复旧系统绑定和旧 `State.GlobalHotkeys`。

### 中低风险：注销失败会留下“幽灵快捷键”

`TryUnregisterGesture` 失败时会保留 Win32 注册，但上层忽略返回值并继续报告成功；该按键可能仍被系统拦截，却不再执行命令。注销路径

建议注销失败时让整批应用失败，或者直接销毁并重建快捷键专用 `HwndSource`，确保系统注册全部释放。

### 第二项中确认正确的部分

- 启动参数、托盘和全局快捷键最终汇入统一命令路径，没有复制三套业务逻辑。
- 次实例不会释放主实例持有的 Mutex。
- 新建纸片入口最终都经过 `CreatePaper`。
- 快捷键注册新组合失败时，会回滚本轮已经成功注册的组合。
- 正常退出时，每个窗口、托盘和 PowerShell 清理步骤都有独立异常隔离。
- PowerShell 回收包含关闭输入、等待退出、超时后终止进程树，正常退出路径实现合理。

当前最建议先修的是：①退出保存结果可判断；②退出前统一提交编辑内容；③保存失败禁止清理图片。三个改动可以一起解决两项审查中最严重的数据丢失链。



### 第三项：胶囊状态机、焦点与菜单

1. 高风险：拖动中隐藏可能永久卡住排序状态

“隐藏全部”会直接拆除槽位，但 `ClearDeepCapsulePlacement` 没有清理：

- `DeepCapsuleGestureState.Reordering`
- `Mouse.OverrideCursor`
- 鼠标捕获
- 延迟的显示器刷新

之后 `ArrangeDeepCapsules` 会一直认为仍在拖动而直接返回。通过全局“隐藏全部”快捷键可以触发，重新显示后胶囊可能不再排布，直到重启。

建议增加统一的 `CancelDeepCapsuleReorderDrag()`，所有隐藏、关闭模式、销毁槽位路径都必须先调用。

1. 中风险：快速反向折叠/展开仍会跳变

中断动画时虽然读取了当前 Width、Height 和透明度，但随后新动画仍从固定的胶囊/纸片端点和固定透明度开始：

- 中断处理
- 重新使用固定起点

因此快速点击展开、Escape 收回、再展开时，仍可能瞬间跳到完整端点再继续动画。

1. 中风险：右键时边缘胶囊仍会缩回

`MouseLeave` 无条件调用 `SetDeepCapsuleHover(false)`；菜单打开状态只用于置顶，没有参与 `SetDeepCapsuleHover`。

鼠标从胶囊移入独立的右键菜单窗口时，胶囊仍会滑回边缘。这与更新日志顶部“右键边缘胶囊不应该收起”的待办一致。

一次 Escape 提交并收回、图片菜单关闭后恢复编辑焦点，以及拖动预览不直接修改持久顺序，这几条主路径没有发现同等级问题。

### 第五项：待办、关联笔记与脚本胶囊

1. 中风险：关联状态改变后，胶囊资格存在多条漏刷新路径

开启“已关联笔记不显示为胶囊”后：

- A 改绑到 B 时，只刷新 B，不刷新 A。
- 空待办按 Backspace 删除时，完全不刷新被解除关联的笔记。
- Undo/Redo 恢复 `LinkedNoteId` 后也不重新计算胶囊资格。

结果可能是笔记一直被隐藏，或者撤销后本应隐藏的笔记仍留在胶囊队列。

建议把关联笔记 ID 的前后差集计算集中到统一的待办变更入口，不要由每条操作路径自行刷新。

1. 中风险：多行粘贴可能需要两次撤销

多行粘贴已经先执行一次 `PushUndoSnapshot`，但它随后修改当前 TextBox 并重建所有行。按 WPF 正常焦点顺序，旧编辑框的 `LostFocus` 会再添加一份“旧首行＋已新增其余行”的混合快照。

表现可能是：

- 第一次 Ctrl+Z 只恢复首行；
- 第二次 Ctrl+Z 才移除新增行。

建议在程序化粘贴和重建期间抑制 LostFocus 快照，或重建前清空当前文本编辑跟踪状态。

常驻 PowerShell 的正常退出链路没有发现明显问题：关闭输入、超时杀进程树、单步清理异常隔离均已实现。

### 第六项：Markdown、图片与外部编辑

1. 高风险：图片限制没有真正约束解码内存

目前图片会先 `ReadAllBytes`，再通过 `BitmapCacheOption.OnLoad` 完整解码，之后才检查尺寸。

另外位图缓存只限制为 96 个条目，不限制解码后的总字节数。连续缩放可为同一图片产生多个宽度缓存；4096×4096 位图单张就可能占约 64MB。

建议：

- 读取前增加输入文件大小硬上限；
- 先读取元数据和像素数，再按目标尺寸解码；
- 缓存改成按解码后字节数控制的 LRU，并对宽度分桶。

1. 中风险：代码块里的图片语法会被错误当成真实资产

渲染器会排除 fenced code block，但 `MarkdownImageReferences.Enumerate` 只是逐行解析，不识别代码块。

因此代码样例中的：

```
![image](i:001)
```

仍可能被跨笔记克隆、外部导出改写，并阻止对应资产清理。应让资产枚举和渲染使用同一套代码块判定。

1. 中风险：外部图片导出会静默降级，临时文件也不会清理

`ConvertMarkdownForExternalEditor` 在目录创建或单张图片写入失败时，会保留内部 `i:` 地址；随后 `WriteExternalMarkdownFile` 仍照常打开文件，普通 Markdown 编辑器只会看到损坏图片。

同时 `%TEMP%\PaperTodo` 下的 Markdown 和图片目录没有过期清理，删除笔记后也会长期残留。

建议让导出返回结构化成功/失败结果，使用临时目录完整写完后再替换，并增加按时间清理旧导出的启动任务。

纯图片 Ctrl+V、图片右键焦点恢复、删除后撤销、跨笔记粘贴时克隆图片资产，这些主要路径本身实现合理。

结论：第四项暂时不能判定通过。基于最新提交 `f8b9c06`，排除初审已经提过的非标准 Snap Layout 识别不足、最大化折叠未先切回 `Normal`、透明无边框架构问题后，仍发现 4 个实质问题。

### 1. 高：断开显示器或跨 DPI 展开时，可能恢复到错误位置甚至屏外

折叠时只保存一个 `_collapsedFromSnappedBounds: Rect?`。这个矩形属于捕获时窗口所在显示器的 DPI 坐标空间，但没有同时保存显示器 ID、捕获 DPI 或格子比例。

点击边缘胶囊时，代码会：

1. 先把主窗口移动到胶囊队列所在显示器；
2. 再把旧矩形直接写入当前窗口的 `Left/Top/Width/Height`。

相关路径见 捕获并直接恢复旧矩形、展开前先移动到胶囊显示器 和 矩形使用当前窗口 DPI 转换。

可触发场景：

- 副屏分屏贴靠 → 折叠 → 断开副屏 → 胶囊回落主屏 → 点击展开：旧副屏坐标未经校验直接恢复，窗口可能重新跑到屏外。
- 胶囊队列在 100% 主屏，纸片被拖到 150% 副屏贴靠：折叠后从主屏胶囊展开，副屏矩形会在主屏 DPI 上重新解释，位置和尺寸可能错位。

Windows 官方也明确说明跨显示器时窗口 DPI 会变化，应用需要基于新 DPI 的建议矩形重新计算位置和大小，而不能复用旧坐标。[WM_DPICHANGED 文档](https://learn.microsoft.com/en-us/windows/win32/hidpi/wm-dpichanged)

建议把运行时字段改为类似：

```
SnapRestoreState {
    MonitorDeviceId,
    PhysicalVisibleRect,
    CapturedDpi,
    HorizontalFraction,
    VerticalFraction,
    TileKind
}
```

恢复时重新查询原显示器工作区；原屏不存在就按格子比例映射到回落屏，而不是直接写旧 `Rect`。

### 2. 中：重启后完全丢失分屏位置

贴靠矩形只存在于 `PaperWindow` 的内存字段中，PaperData 没有任何 Snap 状态。窗口处于贴靠状态时，UpdateGeometry 会直接返回，退出又只是保存现有 `PaperData`。

因此：

> 浮动位置 A → 分屏贴靠 B → 退出 → 重启

纸片会回到旧浮动位置 A，而不是退出时的分屏格子 B。

这不算 v2.2 那句“折叠/隐藏后恢复”的直接违约，但属于第四项“重启恢复”目标的明确缺口。应将上面的 `SnapRestoreState` 持久化，并在显示器或工作区变化时重新解析。

### 3. 中：恢复的是“同样大小的矩形”，不是真正的 Windows 贴靠状态

`RestoreSnapTilePresentation` 仅设置 WPF 的 `Left/Top/Width/Height`，然后修正可见边框；没有让 Windows Shell 重新把窗口加入 Snap 布局。恢复实现

所以当前实现能够保证的只是“视觉上回到格子”，不能保证：

- 继续参与原 Snap Group；
- 拖动相邻窗口分隔线时联动缩放；
- 在任务切换中作为原分屏组合恢复；
- Snap Assist 仍认为它属于原布局。

微软的定义中，窗口需要被系统放入布局区域，再由 Snap Assist 构建完整布局。[Windows Snap Layouts 文档](https://learn.microsoft.com/en-us/windows/apps/desktop/modernize/ui/apply-snap-layout-menu)

如果无法保留原生 HWND 的 Shell 贴靠状态，更新日志建议改成：

> “恢复到此前分屏格子的尺寸和位置”

不要称为完整恢复 Windows 分屏贴靠状态。

### 4. 中：切换主显示器会合并两个物理显示器的胶囊队列

代码把“当前主显示器设备名”归一化成空字符串：NormalizeQueueMonitorDeviceName。

假设：

- 显示器 A 原来是主屏，A 队列保存为 `""`；
- 显示器 B 队列保存为 `\\.\DISPLAY2`；
- 用户把 B 改成主屏。

此时 A 的空标识会解析到 B，而 B 的显式标识也会被归一化为空，两条队列立即合并。下次启动时，StateStore 还会把这个结果写回纸片状态，原物理显示器身份随之丢失。

应始终持久化真实显示器 ID；“断屏临时回落到主屏”只应是布局阶段的临时映射，不能修改或归一化持久化身份。

### 建议修复顺序

1. 先引入带显示器和 DPI 信息的 `SnapRestoreState`，同时解决断屏、跨 DPI 和重启恢复。
2. 将胶囊队列改为永久保存物理显示器身份。
3. 明确区分“视觉矩形恢复”和“Windows 原生 Snap 状态恢复”，相应修改实现或更新日志措辞。
